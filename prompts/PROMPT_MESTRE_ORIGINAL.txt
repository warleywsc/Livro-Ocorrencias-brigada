# Projeto: Livro Digital da Brigada de Incêndio

## Objetivo geral

Estou desenvolvendo a ideia de um app web chamado **Livro Digital da Brigada de Incêndio**, inicialmente para prototipação visual no Google Stitch e, futuramente, para possível implementação dentro do ambiente Microsoft 365 da empresa, preferencialmente usando:

* Power Apps
* Microsoft Lists / SharePoint Lists
* Power Automate
* Biblioteca de documentos SharePoint
* Power BI
* Possível migração futura para Dataverse

O objetivo é substituir o livro físico/manual atualmente preenchido pela empresa contratada da Brigada de Incêndio.

O app deve permitir o registro de:

* Plantão
* Equipe
* Conferência de viaturas e materiais
* Inspeção de rotina
* Ocorrências
* Pendências operacionais
* Procedimentos revisados
* Confirmação de ciência
* Retificações
* Histórico
* Relatórios e indicadores

---

O modelo atual do livro físico tem campos como:

A inspeção de rotina contém itens como:

Por isso, o app não deve ser apenas um formulário digital grande. Ele precisa funcionar como:

**Livro de passagem de serviço + controle de pendências operacionais + registro estruturado de ocorrências.**

# Direção arquitetural recomendada

A melhor solução dentro do Office 365 é uma solução Power Platform composta:

## Recomendação inicial

Usar:

* **Power Apps Canvas**, chamado em português de **Aplicativo de Tela** ou tipo **Tela**
* **Microsoft Lists / SharePoint Lists** como base inicial de dados
* **Power Automate** para notificações, aprovações, retificações e ciência de procedimentos
* **Biblioteca de documentos SharePoint** para anexos, fotos e relatórios
* **Power BI** para indicadores
* **Grupos Microsoft 365 / SharePoint / controle interno de perfis** para permissões

## Por que começar com Lists

Começar com Microsoft Lists é mais simples e mais compatível com ambiente corporativo já disponível.

Não começar com uma única lista gigante.

Usar listas separadas por assunto para facilitar manutenção e possível migração futura para Dataverse.

## Migração futura para Dataverse

É possível começar com Lists e migrar para Dataverse futuramente.

Mas é importante entender:

* Migrar os dados é mais fácil.
* Migrar o app pronto exige ajustes em conexões, fórmulas, filtros, permissões e relacionamentos.
* Para facilitar a migração, as listas devem ser estruturadas desde o início como se fossem futuras tabelas.

Boas práticas para migração futura:

* Criar listas separadas por assunto.
* Usar nomes de colunas sem acento e sem espaços.
* Criar IDs textuais próprios, como `CodigoPlantao`.
* Evitar depender apenas do ID automático do SharePoint.
* Evitar colunas complexas demais.
* Guardar anexos em biblioteca separada.
* Criar lista de perfis desde o começo.
* Evitar um formulário monolítico.

# Regras principais do sistema

1. O líder preenche os dados do plantão.
2. Enquanto o livro estiver em rascunho, pode ser editado.
3. Após finalizado, o livro deve ficar bloqueado para edição comum.
4. Alterações posteriores devem ser feitas por retificação.
5. O sistema deve registrar usuário, data e hora das ações importantes.
6. O próximo turno deve visualizar automaticamente as pendências abertas.
7. O sistema deve diferenciar ocorrência nova de pendência que permanece.
8. Pendências normalizadas devem ter registro próprio.
9. O sistema não deve depender de texto corrido quando for possível estruturar em campos.
10. Deve haver anexos/fotos quando necessário.
11. Deve haver filtros por data, turno, unidade, local, tipo, status, responsável e prioridade.

# POTs Revisados e confirmação de ciência

O item **5.0 POTs Revisados** refere-se aos **Procedimentos de Operação e Testes** relativos à área de incêndio.

Quando houver procedimento revisado ou atualizado, o líder registra a sigla do procedimento para que os próximos turnos tomem ciência e consultem o procedimento.

O app deve ter módulo específico para:

**POTs Revisados e Ciência de Procedimentos**

Campos principais:

* Sigla do procedimento
* Título/assunto
* Unidade/área relacionada
* Data da revisão
* Versão/revisão
* Observação do líder
* Link ou anexo do procedimento
* Registrado por
* Data/hora de registro
* Status

Regra de ciência:

* O botão **Confirmar ciência** só deve aparecer se o usuário logado ainda não confirmou ciência daquele procedimento.
* Se o usuário já confirmou, o botão não aparece.
* No lugar, mostrar: **Ciência registrada em [data/hora].**
* A ciência deve ser armazenada em lista própria.
* Registrar nome, login/e-mail, perfil, data/hora e procedimento confirmado.

Supervisores e gerentes devem conseguir ver:

* Quem confirmou ciência
* Quem está pendente
* Percentual de ciência
* Última confirmação
* Filtro por procedimento, unidade, perfil e status
